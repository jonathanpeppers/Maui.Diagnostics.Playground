## Systemic findings

1. **CoreCLR emits a structured crash report; Mono does not.** For every runtime/native crash that has a managed context (`runtime-access-violation`, `runtime-failfast`, `mixed-managed-native`, native aborts), CoreCLR produced a `DOTNET` crash report **and** scenario-runner managed frames, giving full processв†”logcat agreement. The same scenarios on Mono captured the native tombstone/backtrace but no managed correlation (graded logcat-only). This is the single biggest diagnostic-quality differentiator and lifts access-violation, failfast, and mixed-managed-native from B to A on CoreCLR.
2. **Native crash capture is uniformly strong.** SIGABRT/SIGSEGV/SIGILL scenarios produced tombstones, backtraces, and BuildIds in all four cells via the `CrashNativeKit` interop вЂ” independent of runtime and configuration.
3. **Stack overflow is the hardest case on every runtime.** Mono yields a SIGSEGV with no tombstone; CoreCLR converts it to a SIGABRT with a full tombstone + report but cannot surface the managed `StackOverflowException`. No cell scored above C.
4. **Release strips the vendor-handled breadcrumb.** `ManagedCrashScenarioRunner.CaptureHandledException` logs via `System.Diagnostics.Debug.WriteLine`, which is compiled out of Release builds вЂ” so `vendor-handled-exception` drops from A (Debug) to C (Release) on both runtimes. App-code issue, reproducible and expected.
5. **The on-screen runtime chip is wrong, and so is the analyzer's runtime detection.** `DiagnosticsSelfReportService` uses `Type.GetType("Mono.Runtime")`, which is always `null` on modern MonoVM, so the landing-page chip mislabels the runtime. Likewise `analyze-crash.cs` keys `DetectedRuntime` off `MonoDroid` log tags + `libmonodroid` (present in **both** runtimes), so it always reports `mono`; on CoreCLR cells this fails the non-required `runtime-matches-request` check and costs lifecycle scenarios one bonus point (D vs C). Neither reflects a real diagnostics gap вЂ” ground truth is the loaded native lib.
6. **resource-ui-hang never produced an ANR** within the capture window on API 34 in any cell вЂ” a harness timing limitation rather than a runtime behavior.

## Recommendations

- **Fix runtime self-detection** in `DiagnosticsSelfReportService` (use `RuntimeInformation`/`RuntimeFeature` or probe for `libcoreclr` vs `libmonosgen`) so the on-screen chip is truthful, and update `analyze-crash.cs` `DetectedRuntime` to read the loaded native library (or accept the requested runtime) instead of the always-present `MonoDroid` tag. This alone removes the artificial D grades on the lifecycle scenarios.
- **Make the vendor-handled path observable in Release** вЂ” replace `Debug.WriteLine` in `CaptureHandledException` with a real logger / vendor breadcrumb (`ILogger`, `Android.Util.Log`, or the actual vendor SDK call) so the handled-exception scenario behaves identically in Debug and Release.
- **Harden the ANR scenario**: extend the `resource-ui-hang` settle window past Android's ANR threshold (в‰Ґ5 s blocked + grace) or assert on the watchdog/`ANR in` line so the required artifact is reliably captured.
- **Prefer CoreCLR for production diagnostics** where the crash report matters: it correlates managed and native state on runtime/mixed crashes that Mono leaves as native-only tombstones.
- **Resolve the offline-SDK host pack** (`Microsoft.NETCore.App.Runtime.Mono.win-x64`) to enable a true AOT Release matrix; current Release results are JIT + trimmed only. Re-run the two Release cells with AOT once the pack is available to confirm the trends hold.
